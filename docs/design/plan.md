# 个人科研 AI 系统概念设计报告

详细技术设计已拆分到 [docs/design/README.md](./docs/design/README.md)。若本文与 `docs/design` 在对象定义、状态命名或事件语义上存在差异，以 `docs/design` 为准；本文中的 `PlannerAgent`、`DeveloperAgent`、`ReviewerAgent` 等名称应理解为逻辑角色，而不是固定进程。

## 1. 文档定位

本文档定义一个面向个人科研与小规模协作的 AI 科研操作系统概念方案。目标不是先做一个功能分散的脚本集合，而是定义一个可以长期演进的 Go 控制平面：首版部署简单、日志清晰、可在多机环境运行，后续可逐步扩展为更强的多 agent、多模型、跨集群科研自动化系统。

本文档同时承担三类作用：

- 作为后续实现的总设计基线。
- 作为功能取舍和架构收敛的依据。
- 作为未来生成更细实现文档、接口文档、任务拆解文档的上位输入。

## 2. 设计目标

### 2.1 核心目标

系统需要满足以下目标：

1. 支持长期科研闭环。
   从提出猜想、创建任务、修改代码、执行实验、评估结果、形成报告，到合并成果，形成可持续运行的闭环。
2. 支持长期运行。
   单个任务可能持续数小时、数十小时甚至数天，系统必须具备状态持久化、重启恢复和异常上报能力。
3. 支持多 agent 编排。
   在复杂任务中允许规划、开发、实验、评审等角色分工；在简单任务中允许快速、低成本地直接回答。
4. 支持人工监督与中途干预。
   用户可以通过飞书等渠道接收实时进度、查看阻塞、批准关键动作，并在执行中改变方向。
5. 支持多机与异构环境。
   系统需要能管理本机、SSH 远端机器，以及 SLURM 集群等执行环境。
6. 强调成本可控。
   能根据任务复杂度在不同模型间路由，限制 token、GPU、运行时长和重试次数。
7. 保持部署简单。
   首版优先单二进制部署，减少外部依赖，降低维护门槛。
8. 强调失败上报。
   遇到不会做、做不了、外部故障或环境异常，必须及时生成结构化报告并通知用户。

### 2.2 非目标

以下内容不作为首版强承诺：

- 首版不做完整微服务拆分。
- 首版不强依赖 Temporal、NATS、Postgres 等重型基础设施。
- 首版不承诺全自动自我更新。
- 首版不追求全组织级权限和多租户管理。
- 首版不追求全自动科学结论判断，人工审批仍保留在关键环节。

## 3. 设计原则

### 3.1 单主控优先

首版采用一个 Go 主程序作为控制平面，负责调度、持久化状态、日志、通知、策略和外部适配。这样部署形态简单，便于在个人机器、工作站或服务器上快速落地。

### 3.2 远端执行与主控分离

多机器是首版刚需，但不采用多主控架构。系统通过统一执行器抽象接入本地命令、SSH 远端命令和 SLURM 作业系统。执行能力分散，控制能力集中。

### 3.3 工作流与推理分离

长期状态由主控内建工作流状态机保存，agent 只负责短时决策和动作生成，不负责长期状态托管。

### 3.4 工具与策略分离

Git、飞书、模型、SSH、SLURM、MCP 等能力都以适配器形式接入。适配器只负责“能做什么”，控制平面负责“何时做、是否做、出了问题怎么办”。

### 3.5 薄核心原则

主控核心必须尽量小。核心代码只保留以下职责：

- 工作流状态推进。
- 任务调度与队列。
- 预算与权限裁决。
- 日志、审计与通知。
- 适配器注册与调用编排。

以下能力默认不写进核心：

- 具体 SSH 命令细节。
- 具体 Git 平台交互细节。
- 具体 SLURM 命令封装。
- 具体模型厂商调用细节。
- 具体科研领域工具逻辑。

这些能力应优先放入独立适配器或独立 MCP 服务，核心只依赖稳定接口。

### 3.6 失败视为常态

模型超时、网络波动、集群存储故障、凭据失效、仓库冲突、远端节点掉线都视为正常情况。系统必须内建重试、降级、阻塞、终止和上报策略。

### 3.7 Git 驱动科研

科研过程应围绕 Issue、分支、提交、PR/MR、实验报告组织，保证每一轮迭代都具备可追踪的代码和文档产物。

### 3.8 上下文简短但足够

文档、agent 和任务传递的上下文必须压缩，只保留目标、约束、当前状态、可用工具、最近关键结论和下一步动作，避免把整个历史原样灌入模型。

### 3.9 自然语言唯一用户入口

后续系统默认假设：对人类而言，自然语言是唯一操作入口。CLI、飞书等只是自然语言承载通道，而不是独立的命令语法体系。

配置文件仍然保留，但它属于静态基线和安全边界，不作为日常任务交互入口。用户可以通过自然语言调整运行时设置，例如定时任务、通知策略、观察目标和预算软阈值；但凭据、根目录、MCP 地址和硬安全策略不能被自然语言直接改写。

## 4. 总体架构

### 4.1 首版物理形态

首版系统采用“自然语言唯一用户入口 + 单主控 + 远端执行”架构：

- 一个 Go 单二进制主程序常驻运行。
- 所有人类请求先以自然语言进入，再被解释为结构化意图。
- 自然语言意图识别必须由模型完成，关键词匹配、前缀匹配或正则规则不能作为主判定路径。
- 主程序内置自然语言解释层、CLI/API 通道、工作流状态机、任务队列、日志系统、通知系统和适配器注册层。
- 多台机器通过 SSH MCP、轻量执行器或统一执行协议接入。
- HPC 资源通过 SLURM MCP 或 SLURM 执行适配器接入。
- 所有任务状态和事件由主控统一记录与裁决。

### 4.2 六个子系统

系统在逻辑上拆为六个子系统：

1. `control-plane`
   统一入口，处理命令、事件、定时任务、项目生命周期、审批和通知。
2. `workflow-engine`
   持久化科研状态机，跟踪长期任务、重试、阻塞、人工信号和恢复。
3. `agent-orchestrator`
   管理多 agent 分工、模型能力分类、任务拆解和上下文打包。
4. `executor-adapters`
   提供本地执行，以及对 SSH/SLURM/MCP 执行能力的统一封装。
5. `integration-adapters`
   接入 Git、飞书、模型、调度器，以及独立 MCP 服务。
6. `observability-governance`
   负责结构化日志、审计、预算、权限、健康检查和异常上报。

### 4.3 架构分层示意

```text
Human / CLI / Feishu
        |
   Control Plane
        |
 +------+------+------------------+
 |             |                  |
Workflow   Agent Orchestrator   Governance
  |             |                  |
  +------ Integration Adapters -------------------+
                 |                |               |
              Model API        Notifier        MCP Clients
                                                  |
                              +-------------------+------------------+
                              |                   |                  |
                           Git MCP             SSH MCP           SLURM MCP
                              |
                        Executor Adapters
                              |
                           Local Exec
```

## 5. 首版架构决策

### 5.1 技术栈

建议首版使用以下技术方向：

- 语言：Go
- 部署：单二进制
- 配置：YAML 或 TOML
- 本地持久化：SQLite 或 BoltDB 级别嵌入式存储
- 日志：JSON 结构化日志
- 模板与内置资源：Go `embed`
- 外部通信：HTTP + Webhook + MCP + 命令行调用

这样可以把部署收敛为：

1. 下发一个二进制。
2. 提供一个配置文件。
3. 准备凭据和工作目录。
4. 直接启动。

### 5.2 不引入重型依赖的原因

首版不直接引入 Temporal/NATS 的原因不是否定其价值，而是当前优先级是：

- 快速落地。
- 容易维护。
- 降低外部服务运维复杂度。
- 先把科研闭环真正跑通。

文档需要保留未来迁移位，使后续能够把内建事件队列和工作流引擎替换为独立组件，而不破坏主接口。

### 5.3 MCP 优先的外部能力解耦

首版虽然可以在进程内先落少量适配器，但架构上必须优先按“MCP 优先，内嵌适配器兜底”设计：

- Git 操作应优先抽象为独立 Git MCP。
- SSH 操作应优先抽象为独立 SSH MCP。
- SLURM 操作应优先抽象为独立 HPC/SLURM MCP。
- 领域专用工具优先通过独立 MCP 提供。

这样可以把高变化、高环境耦合的能力从主控核心剥离出去。主控只保留协议、路由、重试和治理逻辑。

### 5.4 自我更新策略

自我更新不纳入首版承诺范围。首版只在架构中预留 `Updater` 接口和版本元数据字段，未来若启用，应采用“受控升级”模式：

- 更新来源可配置。
- 必须做签名校验。
- 默认需要人工批准。
- 支持回滚。

## 6. 核心对象模型

为保证后续能直接落代码，文档定义一组最小公共对象。

### 6.1 ProjectSpec

`ProjectSpec` 表示一个科研项目的主配置，应至少包含：

- `project_id`
- `name`
- `goal`
- `repositories`
- `default_branch`
- `execution_targets`
- `budget_policy`
- `notification_policy`
- `approval_policy`
- `artifact_policy`
- `workspace_policy`
- `memory_policy`
- `evaluation_policy`

其职责是定义项目的目标、代码位置、可用机器、预算、通知规则和工作区治理边界。

### 6.2 TaskSpec

`TaskSpec` 表示一项可调度任务，应至少包含：

- `task_id`
- `project_id`
- `task_type`
- `priority`
- `goal`
- `dependencies`
- `expected_outputs`
- `allowed_models`
- `resource_limits`
- `target_executor`
- `retry_policy`
- `acceptance_policy`
- `memory_hints`
- `trigger`

任务既可以是简单问答，也可以是长时间实验、代码修改、报告生成或定时巡检。

### 6.3 RunRecord

`RunRecord` 表示一次任务运行记录，应至少包含：

- `run_id`
- `task_id`
- `workflow_id`
- `state`
- `timeline`
- `cost_summary`
- `log_index`
- `artifact_refs`
- `failure_reason`
- `intervention_history`
- `result_summary`
- `routing_decision`
- `classifier_summary`

`RunRecord` 是恢复、审计、上报和阶段性报告生成的核心依据。

### 6.4 Event

`Event` 是系统内部统一事件模型，应至少包含：

- `event_id`
- `event_type`
- `source`
- `target`
- `project_id`
- `task_id`
- `severity`
- `payload`
- `attempt`
- `created_at`

事件既用于系统内部协调，也用于形成审计链路。

### 6.5 MemoryRecord

`MemoryRecord` 用于记录可长期复用的信息，应至少包含：

- `memory_id`
- `scope`
- `project_id`
- `repository_id`
- `memory_type`
- `summary`
- `content_ref`
- `source`
- `confidence`
- `freshness`
- `last_verified_at`
- `tags`

建议至少支持三种作用域：

- `global`
  全局偏好、系统经验、通用失败模式、用户习惯。
- `project`
  项目目标、评测标准、常见坑、路线结论、环境约束。
- `task`
  某轮任务中暂时有用、但未必值得长期保留的工作记忆。

记忆类型建议至少覆盖：

- `preference`
- `fact`
- `decision`
- `failure_pattern`
- `playbook`
- `evaluation_baseline`

## 7. 核心接口定义

### 7.1 Executor

执行器负责抽象不同运行环境，最小接口为：

- `Prepare`
- `Run`
- `Query`
- `Cancel`
- `CollectArtifacts`

首版执行层分两类实现：

- `LocalExecutor`
- `RemoteExecutor`

统一执行语义要求：

- 能准备独立工作目录。
- 能执行命令或提交作业。
- 能查询运行状态。
- 能取消任务。
- 能拉取日志和产物。

其中：

- `LocalExecutor` 用于主控所在机器直接执行。
- `RemoteExecutor` 不直接绑定某一种远程协议，而是通过 SSH MCP、SLURM MCP 或未来其他远端执行 MCP 统一驱动。

这样可以避免核心代码里同时堆积大量 SSH、SLURM 和平台特定逻辑。

### 7.2 GitProvider

Git 提供方接口至少包含：

- `CreateIssue`
- `CreateBranch`
- `CommitAndPush`
- `OpenPR`
- `QueryPR`
- `CommentPR`
- `SetCheckStatus`
- `MergePR`

设计要求：

- 接口保持平台抽象。
- 默认由独立 Git MCP 实现。
- 文档示例优先按 GitHub PR 语义撰写。
- 未来兼容 GitLab MR。

其中：

- `CommentPR` 用于回写评测摘要、失败原因和下一轮建议。
- `SetCheckStatus` 用于把自动评测状态暴露到 PR 界面，形成显式门禁。

### 7.3 Notifier

通知接口至少包含：

- `SendProgress`
- `SendAlert`
- `RequestApproval`
- `ReceiveCommand`

首版主通知渠道为飞书。通知至少覆盖：

- 正常进度汇报。
- 阻塞上报。
- 异常告警。
- 人工审批。
- 用户中途指令注入。

### 7.4 TaskClassifier

所有外部任务进入系统后，必须先经过一个低成本、快速响应的 `TaskClassifier`。它的职责不是完成任务，而是做入口分诊：

- 判断是 `fast path` 还是 `task path`。
- 判断是否需要联网。
- 判断是否需要简单执行能力。
- 判断是否需要项目上下文、仓库上下文或记忆检索。
- 给出建议模型档位、预算档位和预估风险。

首版建议 `TaskClassifier` 默认由单 agent 实现，优先使用低价快速模型，例如低成本档的 Codex mini 或 Kimi Code 类模型。原因是：

- 入口分诊必须便宜。
- 入口分诊必须快。
- 入口分诊需要少量工具调用，但不应启动复杂多 agent 编排。

`fast path` 任务至少允许以下能力：

- 联网查询或工具查询。
- 读取少量项目上下文。
- 调用非常简单的执行能力，例如轻量命令、简单脚本、单次 MCP 调用。

如果 `TaskClassifier` 发现任务需要多轮修改、长时运行、较高成本实验、PR 流程或人工审批，则必须转入 `task path`。

### 7.5 ModelRouter

模型路由接口负责根据任务复杂度和预算选择模型，最小能力为：

- 接收任务类型、上下文大小、预算、时限和可靠性要求。
- 接收 `TaskClassifier` 的分诊结果。
- 返回模型选择、执行模式、工具能力需求和回退顺序。

首版要支持两类路径：

- `fast path`
  简单、低风险、即时反馈任务，走低成本快速模型。
- `task path`
  复杂任务进入任务队列和工作流状态机，允许多 agent 分工。

同时还要支持两类模型能力：

- `agentic model`
  如 Codex、Kimi Code 这类自带较强 agent/工具使用能力的模型，适合承担编码、执行、修复、迭代类任务。
- `language-only model`
  如 DeepSeek 一类以语言推理为主的模型，适合承担评审、比较、总结、草拟方案、风险分析类任务。

模型路由不能只看“模型名字”，必须看“能力类别 + 任务类型”的匹配关系。

### 7.6 AgentRuntime

AI 执行层需要定义统一的 `AgentRuntime` 接口，用于屏蔽不同模型的交互差异，至少应支持：

- `Plan`
- `Act`
- `Review`
- `Summarize`
- `ClassifyCapability`

首版建议内建两种运行时：

- `AgenticRuntime`
  面向带 agent 能力的模型，支持执行型任务。
- `LLMRuntime`
  面向纯语言模型，支持评审型和分析型任务。

首版还建议增加一个轻量运行时：

- `FastSingleAgentRuntime`
  面向入口分诊和快速任务处理，允许联网和非常简单的执行动作，但不负责复杂长期编排。

### 7.7 Scheduler

调度接口负责定时任务和周期巡检，至少支持：

- 周期拉起任务。
- 延迟执行。
- 失败重试。
- 错峰调度。

适用场景包括：

- 定时查看学校官网。
- 周期自检。
- 周期拉取集群状态。
- 周期生成项目摘要。

## 8. 工作流设计

### 8.1 状态机

首版内建工作流引擎采用显式状态机，最少覆盖以下状态：

- `Planning`
- `Implementing`
- `Experimenting`
- `Evaluating`
- `Reporting`
- `WaitingHuman`
- `Blocked`
- `Done`
- `Aborted`

### 8.2 状态迁移规则

基本迁移约束如下：

- 由用户创建项目或任务后进入 `Planning`。
- 形成可执行方案后进入 `Implementing` 或 `Experimenting`。
- 实验完成后进入 `Evaluating`。
- 评估完成后进入 `Reporting`。
- 需要人工审批、人工指令或外部决策时进入 `WaitingHuman`。
- 发生可恢复但暂不可继续的问题时进入 `Blocked`。
- 达成目标后进入 `Done`。
- 明确放弃、预算耗尽或任务无继续价值时进入 `Aborted`。

### 8.3 人工干预

工作流必须支持运行时信号：

- 改变任务优先级。
- 暂停或恢复项目。
- 修改研究方向。
- 禁止继续消耗某类资源。
- 强制进入报告阶段。
- 强制终止任务。

用户通过飞书或 CLI 注入的指令必须记录到 `intervention_history`。

### 8.4 PR 驱动的评测闭环

针对“已有代码的模型优化任务”，系统需要支持明确的 PR 驱动评测流程：

1. `DeveloperAgent` 完成一轮代码修改后，通过 Git MCP 创建或更新 PR。
2. PR 打开、更新或打上特定标签后，由 Git webhook 或轮询事件触发 `EvalRequested` 事件。
3. `control-plane` 根据项目中的 `evaluation_baseline` 和任务的 `acceptance_policy` 创建 `EvalTask`。
4. `ExperimentAgent` 在本地、SSH 或 SLURM 环境中执行评测，产出结构化结果。
5. `ReviewerAgent` 或 `LLMRuntime` 读取评测产物，判断是否达标。
6. 若达标，工作流进入 `Reporting`，PR 状态更新为“可审阅”。
7. 若不达标但仍有重试预算，系统发出 `EvalFailed` 事件，自动创建下一轮 `Implementing` 子任务，并把失败摘要、关键指标回退点和建议修复方向写入新任务上下文。
8. 若不达标且重试预算耗尽，工作流进入 `WaitingHuman` 或 `Blocked`，要求人工决策。

这样评测不是附属流程，而是代码优化任务的硬门禁。

## 9. Agent 编排设计

### 9.1 角色划分

首版建议至少保留以下逻辑角色：

- `PlannerAgent`
  负责任务拆解、预算预估、风险识别和下一阶段规划。
- `ResearchAgent`
  负责提出假设、分析实验结果、撰写研究摘要。
- `DeveloperAgent`
  负责代码修改、分支操作、提交和 PR 草案，优先绑定 `AgenticRuntime`。
- `ExperimentAgent`
  负责参数配置、实验执行、集群提交、结果收集。
- `ReviewerAgent`
  负责对结果做交叉检查，判断是否进入下一轮，优先绑定 `LLMRuntime` 或低风险评审模型。

### 9.2 编排原则

- 不是所有任务都需要多 agent。
- 简单任务优先单模型直接完成。
- 复杂任务才进入多 agent 协作。
- 多 agent 输出必须由工作流状态机统一裁决，避免各 agent 自行推进长期状态。
- 编码和执行任务优先分配给带 agent 能力的模型。
- 评审、复核、对比、总结任务优先分配给纯语言模型或低成本模型。

### 9.3 AI 执行层能力分工

AI 执行层在文档中应明确分成两条能力线：

1. 执行型能力线
   面向 Codex、Kimi Code 这类自带 agent 的模型，适合处理代码修改、命令执行、仓库操作、修复迭代等任务。
2. 评审型能力线
   面向 DeepSeek、GLM 或其他纯语言模型，适合处理代码 review、实验结果比较、计划审查、报告润色和风险判断。

两条能力线都通过统一任务模型接入，但调度策略、上下文包和预算策略可以不同。

### 9.4 上下文压缩策略

为解决上下文过长问题，系统需要定义统一的上下文包：

- `Goal`
  当前任务目标和成功标准。
- `Constraints`
  预算、时间、可写目录、禁区、审批要求。
- `CurrentState`
  当前状态、最近事件、阻塞点。
- `RelevantArtifacts`
  仅包含本轮必要文件、日志摘要、实验结论。
- `NextAction`
  当前 agent 需要完成的动作。

禁止默认把整个项目历史、完整日志、全部对话和全部文件直接送入模型。

### 9.5 记忆读取与写回策略

Agent 编排层必须接入记忆系统，但不能把记忆系统做成无边界的大杂烩。首版建议：

- 任务开始前只检索与当前目标最相关的全局记忆和项目记忆。
- 任务结束后只把有复用价值的结论写回长期记忆。
- 临时草稿、低置信度猜测和未经验证的结论只进入任务记忆，不直接提升为项目记忆。
- 只有明确影响后续任务成功率的信息，才应从任务记忆晋升为项目记忆或全局记忆。

## 10. 执行层设计

### 10.1 LocalExecutor

适用于本地开发机、工作站和单机任务，特点：

- 调试成本低。
- 适合轻量命令、文档生成和小规模实验。
- 适合作为默认回退执行器。

### 10.2 RemoteExecutor

`RemoteExecutor` 是远端执行统一封装，不直接把 SSH/SLURM 细节硬编码在核心里。它的职责是：

- 根据任务选择合适的远端 MCP。
- 统一执行请求与结果模型。
- 处理超时、重试、取消、日志抓取和产物收集。

首版建议优先支持两种远端 MCP：

- `SSH MCP`
- `SLURM MCP`

适用于多台远端机器和集群环境，特点：

- 主控统一调度，远端负责执行。
- 支持工作目录准备、命令执行、日志抓取、产物回收。
- 支持远端主机标签和能力声明，例如 CPU 型、GPU 型、大内存型。

### 10.3 多机资源建模

系统需维护一个机器或执行目标清单，每个目标至少包含：

- `executor_id`
- `executor_type`
- `host`
- `labels`
- `capabilities`
- `workspace_root`
- `health_status`
- `concurrency_limit`

后续调度策略可以基于标签匹配，例如：

- `gpu:a100`
- `site:ihep`
- `queue:long`
- `storage:fast`

## 11. 集成适配层设计

### 11.1 Git 集成

Git 是科研流程主骨架。首版应优先通过独立 Git MCP 接入，并支持：

- 根据任务自动创建 Issue。
- 按任务创建分支。
- 在阶段结束时提交并推送。
- 生成 PR/MR 草案。
- 将实验摘要附到 PR/MR。

系统应明确约束：

- 每次阶段性改动后必须有提交。
- 长任务恢复后必须校验分支状态。
- 合并前保留人工审批钩子。

### 11.2 飞书集成

飞书是首版主要人机协作入口，至少承担：

- 进度通知。
- 阻塞告警。
- 预算消耗提醒。
- 审批卡片。
- 人工指导指令入口。

对高优先级异常需要支持 `@` 用户通知。

### 11.3 模型接入

系统要支持多个模型提供方并统一治理。首版至少在设计上兼容：

- Codex / OpenAI 系列
- Kimi Code / Kimi
- DeepSeek / GLM 等纯模型

模型接入层只负责调用，不负责业务流程。所有成本、超时、回退、优先级和降级逻辑由 `ModelRouter` 管理。

首版文档必须明确区分两类模型：

- 带 agent 能力的编码执行模型
  例如 Codex、Kimi Code，适合写代码、调用工具、执行修复闭环。
- 纯语言推理模型
  例如 DeepSeek、GLM 等，适合 review、比较、总结、写报告和做次级判断。

系统不要求所有模型都具备执行能力。相反，调度器应利用这种差异做职责分工。

### 11.4 MCP 与领域技能

MCP 和技能系统在首版不应被视为可有可无的附属品，而应作为解耦主手段。推荐优先外置的能力包括：

- Git 操作
- SSH 操作
- SLURM/HPC 操作
- 飞书通知
- 领域专用技能

后续 MCP 可继续用于：

- IHEP 作业提交与查询。
- SSH 能力标准化。
- 历史知识库访问。
- 自动报告与 PPT 生成。
- 科研领域专用工具封装。

### 11.5 记忆系统

记忆系统是首版需要明确设计的横向能力，用于避免每次任务都从零开始。它不应替代主数据库和日志系统，而应服务于“高价值经验复用”。

记忆系统建议拆成三层：

1. 工作记忆
   面向当前任务或当前运行轮次，生命周期短，随任务结束可归档或丢弃。
2. 项目记忆
   面向具体项目，保存项目目标、评测基线、有效路线、已证伪路线、环境约束和常见错误。
3. 全局记忆
   面向整个系统，保存用户偏好、通用经验、常见故障模式、常用操作手册和模型使用策略。

记忆写入原则：

- 不自动把所有日志写成记忆。
- 只沉淀高价值、可复用、可描述的结论。
- 优先保存“为什么这样做”而不是原始长文本。
- 对过时信息要允许降权、失效或重新验证。

记忆读取原则：

- `TaskClassifier` 可以读取少量全局记忆，用于识别用户偏好、默认执行风格和常见约束。
- `PlannerAgent` 和 `ResearchAgent` 优先读取项目记忆，用于避免重复试错。
- `DeveloperAgent` 可读取项目记忆中的仓库约定、评测门槛和历史失败模式。
- `ReviewerAgent` 可读取历史评测基线和过去失败案例，辅助判断这次是否真的改进。

记忆来源建议包括：

- 用户显式说明的偏好和规则。
- 人工批准或否决留下的决策记录。
- 反复出现的失败模式。
- 已验证有效的实验路线。
- PR 评测闭环中沉淀出的评测基线和阈值。

### 11.6 MemoryManager

为避免记忆系统和业务代码耦合，建议定义 `MemoryManager` 接口，至少包含：

- `Recall`
- `WriteTaskMemory`
- `PromoteToProjectMemory`
- `PromoteToGlobalMemory`
- `DecayOrArchive`
- `AttachMemoryContext`

`MemoryManager` 应负责：

- 按作用域检索。
- 按置信度和新鲜度排序。
- 控制写回阈值。
- 防止重复、冲突和过期记忆污染后续任务。

## 12. 日志、审计与观测

### 12.1 日志设计目标

日志必须满足四个要求：

- 人能看懂。
- 机器可检索。
- 出问题能回放。
- 多项目并行时能快速定位。

### 12.2 统一日志字段

首版结构化日志至少包含以下字段：

- `timestamp`
- `level`
- `project_id`
- `task_id`
- `workflow_id`
- `agent_id`
- `executor_id`
- `event_type`
- `attempt`
- `cost`
- `duration`
- `result`
- `message`

### 12.3 日志输出策略

- 控制台输出给人读的摘要日志。
- 文件或本地存储保留 JSON 全量日志。
- 核心状态迁移必须形成事件日志。
- 所有失败必须记录机器可解析的错误码或错误类别。

### 12.4 可回放性

回放的最小目标是：

- 能看到任务进入了什么状态。
- 能看到调用了哪些外部系统。
- 能看到每次失败与重试原因。
- 能看到最终为什么阻塞、终止或完成。

## 13. 成本治理与权限治理

### 13.1 预算体系

预算体系至少分三层：

- 项目级预算
  控制项目总 token、总 GPU 时长、总运行时长、总重试次数。
- 任务级预算
  控制单任务可消耗的模型调用、实验资源和最长执行时间。
- 动作级预算
  控制单次模型调用、单次远端执行、单次集群提交的边界。

### 13.2 模型路由策略

模型路由至少考虑：

- 任务复杂度
- 上下文大小
- 响应时限
- 预算余量
- 可靠性要求
- 是否需要代码修改或长链推理

### 13.3 权限治理

首版权限边界必须写清：

- 允许写入哪些目录。
- 禁止写入哪些目录。
- 哪些操作必须审批。
- 哪些凭据仅能保存在服务器侧或本地安全配置中。

默认需审批的动作包括：

- 合并 PR/MR。
- 删除重要产物。
- 大规模资源申请。
- 高风险远端命令。
- 将系统升级到新版本。

## 14. 工作区治理

### 14.1 目录策略

系统需要定义明确的目录边界：

- `workspace/`
  项目工作区和仓库检出目录。
- `artifacts/`
  结果产物、报告、图表和实验输出。
- `state/`
  工作流状态、事件索引和运行记录。
- `logs/`
  结构化日志。
- `tmp/`
  临时文件和可清理内容。

### 14.2 清理策略

- 临时目录可按策略自动清理。
- 失败任务的现场信息必须在清理前归档摘要。
- 长期产物需归档，不允许散落在 home 目录。

### 14.3 提交约束

每个阶段结束后系统至少应执行以下动作中的一部分：

- 记录阶段结果摘要。
- 产出或更新 Markdown 文档。
- 根据需要 `commit` 与 `push`。
- 将实验结论与下一步建议写入项目记录。

## 15. 异常分级与恢复策略

### 15.1 异常分类

系统将异常分为四类：

1. 可重试故障
   例如模型 API 短时失败、网络抖动、SSH 短断。
2. 可降级故障
   例如高阶模型不可用，退化到较弱模型；远端节点不可用，切到本地或其他机器。
3. 需人工介入故障
   例如权限不足、仓库冲突、预算策略冲突、关键产物缺失。
4. 必须终止故障
   例如数据损坏、核心凭据失效且无法恢复、任务已无继续价值。

### 15.2 默认动作

每类异常的默认动作如下：

- 可重试故障：按退避策略重试，并记录尝试次数。
- 可降级故障：执行预定义回退路径，并标记结果可信度下降。
- 需人工介入故障：进入 `WaitingHuman` 或 `Blocked`，立即飞书通知。
- 必须终止故障：进入 `Aborted`，生成明确终止报告。

### 15.3 “做不了及时上报”

这是系统的硬约束。出现以下情况时，必须主动上报而不是静默失败：

- 工具不会用或调用结果不确定。
- 模型在多次尝试后仍无法完成任务。
- 环境与预期不符。
- 需要额外权限但未获批准。
- 花费大量时间后仍未形成可继续推进的路径。

上报内容至少包括：

- 发生了什么。
- 已经尝试了什么。
- 为什么当前做不了。
- 建议用户采取什么动作。

## 16. 部署与运维设计

### 16.1 首版部署目标

首版部署目标是“一个主控、最少依赖、可直接运行”：

1. 准备主配置文件。
2. 准备模型、Git、飞书、SSH 等凭据。
3. 配置工作目录和状态目录。
4. 启动单个 Go 二进制。

### 16.2 运行模式

建议支持两种运行模式：

- 前台调试模式
  便于本地观察日志和手动调试。
- 守护进程模式
  便于长期任务常驻运行。

### 16.3 健康检查

主控应定时执行自检，包括：

- 模型接口连通性。
- Git 凭据有效性。
- 飞书通知通道是否正常。
- SSH 目标是否在线。
- SLURM 命令是否可用。
- 存储目录是否可写。

自检结果应形成健康摘要，并在异常时主动通知。

## 17. 演进路线

### 17.1 v1

`v1` 聚焦把系统真正跑起来，范围包括：

- 单二进制主控
- 内建持久化状态机
- GitHub 优先的 Git 工作流
- 飞书通知和人工干预
- Local 执行器 + MCP 驱动的远端执行
- 结构化日志
- 异常上报
- 预算控制
- 基础多模型路由
- `AgenticRuntime` 与 `LLMRuntime` 双运行时

### 17.2 v2

`v2` 在不推翻接口的前提下增强：

- MCP 适配层标准化
- 多 Git Provider
- 更丰富的模型接入
- 知识记忆库
- 任务模板与科研 skill 体系
- 自动生成阶段性 Markdown 报告
- 自动汇总实验指标

### 17.3 v3

`v3` 面向更强规模和更强治理：

- 可拆分部署
- 外部事件总线和工作流引擎替换位
- 跨机器池调度
- 组织级权限与审计
- 受控自更新

### 17.4 迁移原则

后续增强必须遵守：

- 接口稳定优先于内部重写。
- 事件模型稳定优先于消息实现细节。
- 执行器语义稳定优先于底层调用方式。
- 文档和报告产物必须持续兼容。

## 18. 典型运行示例

这一节用三个典型例子说明系统在实际运行中的路径差异。目的不是给出唯一流程，而是明确不同任务类型应如何被路由、执行和上报。

### 18.1 示例一：简单天气问答

任务示例：

- 用户提问“明天北京天气怎么样，需不需要带伞？”

系统运行方式：

1. 请求从 CLI 或飞书进入 `control-plane`。
2. `TaskClassifier` 先用低成本快速模型做入口分诊，判断该任务属于低风险 `fast path`。
3. 分诊结果要求具备联网能力，但只需要非常简单的执行能力，不需要完整工作流和 PR 流程。
4. 主控将请求交给 `FastSingleAgentRuntime`，默认只允许少量工具调用和轻量执行。
5. 如果配置了天气类工具或外部天气 MCP，则先调用工具获取结构化天气信息。
6. 如果没有工具，则由低成本模型直接回答，并显式标记答案来源是“模型推断”还是“工具查询”。
7. 主控只记录一条轻量 `RunRecord`，包含输入、模型选择、耗时、成本和结果摘要。

这类任务的特点：

- 优先走低成本单 agent。
- 不进入长期状态机主循环。
- 不占用多 agent 编排资源。
- 只保留必要审计日志。

### 18.2 示例二：已有代码的模型优化任务

任务示例：

- 用户要求“在已有仓库中优化某个模型训练脚本，提高推理吞吐或精度”。

系统运行方式：

1. 用户在项目上下文中提交任务，主控创建 `TaskSpec` 和完整 `RunRecord`。
2. `TaskClassifier` 先判断这不是 `fast path`，因为它需要仓库上下文、代码改动、评测和 PR 闭环，因此转入 `task path`。
3. 工作流进入 `Planning`，`PlannerAgent` 先读取仓库摘要、现有指标、预算约束、项目记忆中的评测基线，以及可用执行环境。
4. `ModelRouter` 将核心执行任务分配给 `AgenticRuntime`，例如 Codex 或 Kimi Code，因为它们更适合做代码修改、工具调用和迭代修复。
5. Git 操作通过 Git MCP 完成，包括创建 Issue、创建分支、提交变更和起草 PR。
6. PR 打开或更新后，通过 webhook 触发 `EvalRequested`，由系统自动拉起评测任务。
7. 代码执行与评测通过 `LocalExecutor` 或 `RemoteExecutor` 完成；如果需要 GPU 或远端环境，则通过 SSH MCP 或 SLURM MCP 调度。
8. 训练、评测或 profiling 结果回写为结构化产物，进入 `Evaluating`。
9. `ReviewerAgent` 或 `LLMRuntime` 使用 DeepSeek 这类纯语言模型执行 review，总结以下内容：
   - 修改是否真的命中瓶颈。
   - 指标是否改善。
   - 是否引入行为回退风险。
   - 是否建议继续迭代还是进入报告与 PR。
10. 若结果达标，工作流进入 `Reporting`，自动生成 Markdown 摘要并附到 PR。
11. 若结果不达标但仍有重试预算，则系统自动创建新的 `Implementing` 子任务，并把失败指标、失败原因和建议改写方向送回 `DeveloperAgent`。
12. 若结果不达标且预算耗尽，则进入 `WaitingHuman`，请求用户决定是否继续。

这类任务的特点：

- 是标准 `task path`。
- 执行型模型负责“做”，评审型模型负责“看”。
- Git、SSH、SLURM 不进入核心代码，而通过 MCP 或适配器调用。
- PR 是评测触发器，评测失败会自动回写到下一轮开发任务。
- 每一轮都产生可回溯代码与实验记录。

### 18.3 示例三：没有固定范式的长期科研探索任务

任务示例：

- 用户提出“探索某个新的高能物理问题解法”，没有固定实验模板，也不确定最终采用哪条技术路线。

系统运行方式：

1. 用户创建项目目标后，主控建立长期工作流，而不是一次性任务。
2. 工作流首先进入 `Planning`，由 `ResearchAgent` 和 `PlannerAgent` 联合形成首轮假设、候选路线、预算上限和阶段性停止条件。
3. 主控把探索任务拆成一组子任务，例如文献梳理、复现实验、基线实现、参数扫描、失败归因、阶段报告。
4. 其中：
   - 需要写代码、跑实验、改脚本的子任务优先交给 `AgenticRuntime`。
   - 需要做思路比较、失败分析、路线评估的子任务优先交给 `LLMRuntime`。
5. 当某一轮实验需要远端资源时，通过 SSH MCP 或 SLURM MCP 提交到对应机器或集群。
6. 每一轮结束后，系统必须强制生成阶段总结，至少包括：
   - 当前假设是什么。
   - 哪些被验证，哪些被否定。
   - 花费了多少预算和时间。
   - 下一轮是否值得继续。
7. 如果长期运行中出现连续失败、指标无改进、资源不足或路径不清，工作流进入 `Blocked` 或 `WaitingHuman`，并通过飞书通知用户介入。
8. 用户可以在运行中发出方向修正，例如“停止路线 A，转向路线 B”或“先写一份中期报告”，主控将其作为人工信号写入 `intervention_history` 并重规划。
9. 当探索形成阶段性成果时，系统生成正式 Markdown 报告；若产生代码成果，则再进入 Git 分支和 PR 流程。

这类任务的特点：

- 是长期状态机驱动，而不是单次执行。
- 允许没有固定范式，只要求每轮都形成明确中间结论。
- “做不了及时上报”比“一定做完”更重要。
- 系统重点是持续收敛、记录和请示，而不是盲目长时间空转。

### 18.4 示例四：每天早上 9 点获取新邮件并发送总结消息

任务示例：

- 用户要求“每天早上 9 点拉取新邮件，生成摘要，并把总结发到飞书”。

系统运行方式：

1. 用户为某个项目或全局助手创建一个周期任务，`Scheduler` 记录 Cron 规则，例如 `0 9 * * *`。
2. 到达触发时间后，调度器生成一次新的 `TaskSpec`，其 `trigger` 标记为 `scheduled`，并附带“上次成功拉取时间”或“上次邮件游标”。
3. `TaskClassifier` 将其识别为定时运营型任务，而不是即时问答或长期科研任务。它通常进入轻量 `task path`：
   - 需要联网。
   - 需要外部邮箱读取能力。
   - 需要摘要生成。
   - 需要消息推送。
4. 主控通过独立邮件 MCP 或邮箱适配器拉取“自上次游标以来的新邮件”。
5. 系统先做去重和过滤，例如：
   - 排除已处理邮件。
   - 按发件人、主题或规则分类。
   - 过滤垃圾通知或低优先级自动消息。
6. 若没有新邮件，系统可以：
   - 不发送消息，只记录本次空结果。
   - 或发送一条简短“今日无新邮件”的摘要，取决于通知策略。
7. 若存在新邮件，`FastSingleAgentRuntime` 或低成本摘要模型先生成结构化摘要，至少包含：
   - 重要邮件列表。
   - 每封邮件的一句话摘要。
   - 是否存在需要人工立即处理的邮件。
   - 建议动作，例如“回复”“转发”“忽略”。
8. 若邮件量大或包含复杂附件，可升级到 `task path` 中更强的模型或多步摘要流程，但默认先走低成本摘要。
9. `Notifier` 通过飞书发送每日邮件总结；如检测到高优先级邮件，可额外触发告警消息。
10. 本次处理结果写入 `RunRecord`，并更新邮件游标；若某种分类规则被人工反复修正，可写入全局记忆或项目记忆。

这类任务的特点：

- 由 `Scheduler` 驱动，而不是用户即时触发。
- 需要稳定的去重与游标管理，避免重复总结同一批邮件。
- 通常不需要多 agent，但要允许从低成本摘要升级到更复杂处理。
- 很适合与记忆系统联动，逐渐学会哪些邮件重要、哪些应被忽略。

## 19. 关键验收场景

本设计不以“画出架构图”为完成标准，而以关键场景可落地为标准。

### 19.1 功能场景

至少要验证以下场景：

1. 单项目完整科研闭环。
   从创建任务、开分支、修改代码、执行实验、写报告到生成 PR。
2. 多项目并行调度。
   多个项目同时运行，日志与预算互不混淆。
3. SSH 远端执行。
   任务被分配到远端机器，状态和产物能回收。
4. SLURM 作业提交与查询。
   能提交、查询、取消并把状态映射回统一工作流。
5. 飞书进度与人工干预。
   用户收到进度，能中途要求调整方向。
6. 模型接口短时失败后的重试与降级。
7. 集群存储故障后的阻塞和告警。
8. 预算耗尽后的停止与报告。
9. 代码提交后自动生成 PR 和实验摘要。

### 19.2 非功能场景

至少要验证以下非功能要求：

- 单机部署步骤短且可重复。
- 日志可按项目、任务、状态快速过滤。
- 主控重启后任务可恢复。
- 单个远端节点失联不导致全局崩溃。
- 错误报告能让人快速定位问题。

### 19.3 失败上报验收

当出现连续失败、权限不足、工具缺失、模型不可用时，系统必须：

- 明确说明失败类别。
- 说明已经尝试的恢复动作。
- 给出下一步建议。
- 触发飞书告警或审批。

## 20. 面向后续实现的目录建议

如果后续开始落代码，建议按以下逻辑组织仓库：


## 21. 结论

该系统的首版不应试图一开始就变成完整分布式平台，而应先落地一个足够稳的 Go 单主控：部署简单、状态可恢复、日志清晰、飞书可通知、Git 可追踪、远端机器和 SLURM 可调度、失败会主动上报。

在这个基础上，再逐步引入 MCP、记忆库、领域技能、多提供方兼容、外部工作流和事件总线，系统才能在不推倒重来的前提下自然成长为真正可长期运行的个人科研 AI 系统。
