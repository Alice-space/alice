# 07. 并发、故障与安全设计

## 1. 目标

系统长期运行时，真正难的是并发与边界。该文件定义什么可以并行、什么必须串行、冲突时怎么裁决。

## 2. 并发域

### 2.1 项目级

- 同一项目可以并行存在多个只读任务。
- 同一项目的长期工作流数量可配置。
- 高预算任务之间可设置项目级并发上限。

### 2.2 仓库级

- 同一仓库允许并发只读分析。
- 同一仓库同一分支只允许一个写入型运行持有写锁。
- 不同分支可并行写入，但共享执行环境仍要受资源上限约束。

### 2.3 资源命名空间级

- 每个写入型 `run` 必须独占工作目录。
- 共享缓存、实验输出、GPU 临时目录必须带命名空间或只读挂载。
- 产物目录应视为 immutable，避免后续运行覆盖旧结果。

### 2.4 执行器级

- 每个 `executor_id` 声明自己的 `concurrency_limit`。
- 控制平面必须做本地配额检查，不能把过载问题完全推给远端系统。

### 2.5 评测级

- 同一 PR 的不同评测 shard 可以并行。
- 同一 PR 同一 commit 的重复评测必须幂等。
- 新 commit 到来时，老 commit 的评测应标记为 `Superseded`。

## 3. 锁模型

### 3.1 锁类型

- `project_lock`
- `repo_branch_write_lock`
- `workspace_lock`
- `executor_slot_lock`
- `memory_promotion_lock`
- `schedule_instance_lock`

### 3.2 获取顺序

为避免死锁，获取顺序固定为：

1. `project_lock`
2. `repo_branch_write_lock`
3. `workspace_lock`
4. `executor_slot_lock`
5. `memory_promotion_lock`

不允许逆序获取。

### 3.3 租约与 fencing

- 锁必须有 owner、创建时间、过期时间和 `lease_token`。
- 长任务必须定期续约。
- 每次重新获取锁都必须产生单调递增的 `lease_token`。
- 所有写回、产物收集、PR 状态更新都必须校验 `lease_token`，防止陈旧执行者回写。
- 主控崩溃恢复后要清理僵尸锁并重建租约。

## 4. 幂等性

### 4.1 必须幂等的动作

- webhook 处理
- 定时任务实例创建
- PR 状态更新
- 评测回写
- 人工信号注入
- 通知发送结果补写

### 4.2 幂等键建议

- webhook：平台事件 ID
- 定时任务：`schedule_id + scheduled_time`
- PR 评测：`pr_id + commit_sha + eval_template`
- 人工信号：`workflow_id + seq`
- 通知：`run_id + notification_kind + attempt`

## 5. 典型竞争条件

| 场景 | 风险 | 处理 |
| --- | --- | --- |
| 同一分支两个写入任务同时开始 | 代码互相覆盖 | 分支写锁，后者排队或失败 |
| 不同分支共用同一工作目录 | 工作树污染 | `workspace_lock` + 独立 worktree |
| PR 评测未结束又 push 新 commit | 旧结果污染新状态 | 旧任务标记 `Superseded` |
| 定时任务重复触发 | 重复处理同一批邮件/网页 | 调度幂等键 + 游标去重 |
| 人工终止与自动重试同时发生 | 状态不一致 | 人工信号优先，重试撤销 |
| 记忆晋升与记忆衰减同时发生 | 记忆版本冲突 | 按版本号和 CAS 语义更新 |
| 主控重启后旧执行器仍在写回 | 新旧状态互相覆盖 | `lease_token` fencing |

## 6. 失败模型

### 6.1 按来源分类

- `model_provider_failure`
- `executor_failure`
- `integration_failure`
- `policy_blocked`
- `resource_exhausted`
- `data_or_artifact_inconsistent`
- `terminal_logic_failure`

### 6.2 按处理语义分类

- `retryable`
- `degradable`
- `human_required`
- `terminal`

### 6.3 恢复动作

- `retryable`：有限退避重试。
- `degradable`：切换模型、执行器或工具链。
- `human_required`：挂起并通知。
- `terminal`：终止并产出总结。

规则：

- 失败分类必须同时记录“来源”和“处理语义”，不能只给一个模糊标签。
- `Superseded` 不属于失败。

## 7. 熔断与恢复探测

- 模型供应商、执行器和集成接口都应有独立熔断器。
- 熔断器至少定义：观察窗口、失败阈值、冷却时间、半开探测动作。
- 进入 open 状态后，新任务默认不再路由到该目标，只保留少量探测请求。
- 探测恢复成功后才允许重新放量。

## 8. 预算与降级

- 当项目预算耗尽时，新任务默认不再启动，只保留告警和报告任务。
- 当高成本模型不可用时，可降级到更低成本模型，但不得在高风险评审上静默降级。
- 当执行器连续失败时，应优先切换执行器，再考虑继续重试。

## 9. 人工优先级

人工信号高于自动调度：

- 人工终止高于自动重试。
- 人工改方向高于默认计划。
- 人工批准可突破默认预算，但必须审计。

## 10. 安全约束

- agent 不直接持有系统管理员权限。
- 凭据不出现在模型上下文。
- 远端执行默认最小权限。
- 删除、合并、升级、外部资源申请默认是高风险动作。
- 配置变更提案在批准前不得触发真实环境改动。

## 11. 边界情况清单

- 主控重启时正在运行的远端任务如何接管。
- webhook 晚到、乱序到达。
- 远端任务完成但产物拉取失败。
- 评测通过但回写 PR 状态失败。
- 邮件任务成功拉取但飞书发送失败。
- 记忆命中冲突且无人工确认。
- 半开状态下的熔断器如何防止探测请求放大故障。

这些边界情况都不能靠“后面看日志再说”，必须在实现中成为显式分支。
