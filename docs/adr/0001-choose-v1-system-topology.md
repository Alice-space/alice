# ADR 0001: 选择第一版系统拓扑

- Status: Accepted
- Date: 2026-03-08
- Deciders: Alice repository maintainers
- Related: [`docs/draft.md`](../draft.md)

## 背景

Alice 的第一版需要同时满足几件事：

- 任务状态必须有单一真源，不能把真实状态散落在 issue、PR、评论和各类回调里
- 系统要能恢复、能审计、能解释，不希望一开始就陷入高运维复杂度
- 外部平台能力差异很大，例如 GitHub、GitLab、Cluster 和控制面，权限边界、故障模式和部署方式也不同
- 仓库仍处于早期阶段，当前目标是先把工作流和状态机跑通，而不是一开始就建设完整分布式基础设施

如果一开始把核心运行时拆成多服务，再叠加外部消息队列、数据库和远程 Agent 编排，虽然理论上更“标准”，但会显著提高调试、恢复和演进成本，也会让“状态到底以谁为准”在第一版就变得模糊。

## 决策

第一版采用以下系统拓扑：

1. 使用 Go 作为主实现语言。
2. 将 Alice 核心运行时实现为单个主二进制。
3. BUS 保持为进程内核心组件，内部统一承载：
   - Event Bus
   - State Store
   - 去重
   - outbox
   - scheduler
   - 告警与恢复协调
4. GitHub、GitLab、Cluster、Control 等 MCP 不并入主二进制，而是在同一仓库内分别实现、分别构建，并以独立进程或独立服务运行。
5. Issue、PR、评论和 Webhook 仅作为协作界面、外部事件来源和回执通道，不作为系统真实状态源。

## 选择理由

### 为什么核心运行时先做成单二进制

- 任务状态推进、幂等去重、outbox 写入和恢复流程天然耦合，放在一个进程内更容易保持一致语义
- 第一版最重要的是把状态机、审核、预算、恢复和人工干预闭环跑通，而不是先解决跨服务一致性
- 单进程更容易做本地调试、日志排查、故障复盘和状态恢复
- 在还没有稳定流量模型前，先避免引入额外消息系统和服务治理成本

### 为什么 MCP 必须独立于主二进制

- 不同 MCP 面向不同外部系统，权限、依赖和失败模式不同，独立部署更清晰
- 外部集成不应污染核心运行时边界，避免把 Git 平台、集群控制和内部控制面逻辑揉成一个大进程
- 独立 MCP 更利于最小权限控制、单独升级、单独熔断和单独观测
- 当某个外部平台异常时，核心 BUS 应该还能继续维护真实状态并决定重试、恢复或告警，而不是跟着整个进程一起失稳

### 为什么状态真源必须留在 BUS

- Issue/PR 平台是人类协作界面，不适合作为高保真状态机存储
- 外部 Webhook 是至少一次投递，天然可能乱序、重试、重复；如果把外部平台当真源，恢复和对账会更脆弱
- 把真实状态统一收束到 BUS，才能让审核版本绑定、预算熔断、取消传播和 outbox 恢复有可验证的单一依据

## 被拒绝的备选方案

### 方案 A：第一版直接拆成多个核心微服务

例如把状态机、调度、审核、预算、Webhook 接入分别拆成多个服务，再用消息队列和数据库连接。

不选择的原因：

- 在需求和状态边界尚未完全收敛前，拆分服务会放大接口设计成本
- 跨服务一致性、重试、幂等和故障恢复会提前成为主问题
- 对当前阶段而言，复杂度增加明显大于收益

### 方案 B：把 MCP 直接内嵌进主二进制

例如让 GitHub、GitLab、Cluster、Control 适配器都作为主进程内部模块运行。

不选择的原因：

- 会模糊核心运行时与外部副作用执行器的边界
- 不利于做独立权限控制、独立部署和独立故障隔离
- 某个外部系统依赖异常时，更容易拖垮整个主进程

### 方案 C：把 Issue / PR 平台当作任务状态真源

例如直接通过 issue 标签、评论、PR 状态组合来表达任务真实状态。

不选择的原因：

- 外部平台状态表达能力不足，难以覆盖内部状态机的精细语义
- Webhook 回流并不等于严格一致的事务日志
- 审核轮次、预算状态、恢复子状态、取消传播等内部语义会被外部平台模型严重牵制

## 影响

正面影响：

- 第一版实现路径更聚焦，可以先把核心语义跑稳
- 系统更容易调试、对账、恢复和做审计
- MCP 边界从一开始就清晰，后续可独立扩展外部平台能力

负面影响：

- 主二进制会承载较多职责，需要严格控制模块边界
- 未来若要水平扩展 BUS，需要再补分片、分布式一致性和跨实例恢复方案
- 进程内 BUS 在吞吐和隔离性上不如天生分布式方案，需要靠后续演进逐步补足

## 落地约束

基于该决策，第一版实现应遵守以下约束：

- `cmd/` 下至少区分核心运行时与各 MCP 的独立入口
- 核心 BUS 不直接依赖某个 Git 平台或 Cluster 的具体 SDK 语义作为状态真源
- 所有外部副作用都必须通过 outbox 和 MCP 边界发出，而不是直接改外部状态后再“补写”内部状态
- 任何人机交互界面都只能反映或驱动 BUS 状态，不能绕开 BUS 直接推进主流程

## 后续 ADR

这个 ADR 只回答“第一版系统拓扑是什么”，不展开更细决策。后续至少还需要补：

- 状态存储与恢复策略，例如 JSONL + snapshot 的选择
- MCP 幂等键与 outbox 契约
- 审核租约与聚合策略
- 预算熔断与评测恢复策略
