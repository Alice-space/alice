# ADR 0005: 评测独立于代码审核，预算触发硬熔断并传播取消

## 背景

科研探索和部分工程任务需要重型实验验证。若把评测混入代码审核，或把预算只当展示字段，会导致：

- 代码审核职责膨胀
- GPU/CPU 消耗失控
- 任务回退后旧实验继续运行

## 决策

采用以下策略：

- `evaluation` 是独立状态，不并入 `code_audit`
- 只有 `requires_evaluation` 任务进入评测
- 评测始终绑定 `pr_head_sha + EvalSpec`
- `UsageLedger` 持续累计 token、模型费用和 CPU/GPU 用量
- 预算剩余小于 0 或命中硬上限时，立即进入 `budget_exceeded`
- 触发熔断时传播 `CancellationToken`，停止新的编码和评测，并尽力取消集群作业

## 备选方案

### 方案 A：评测只是审核附加步骤

优点：

- 流程图简单

缺点：

- 难以独立控制预算
- 审核与实验职责混淆

### 方案 B：预算只做告警不做硬限制

优点：

- 实现简单

缺点：

- 容易失控
- 不符合当前 CDR 的强约束

## 影响

- 评测和预算成为状态机一级公民
- 需要独立的账本和 cluster job 对账
- 任务回退、取消或 supersede 时必须做取消传播

## 后续事项

- 明确 token 成本和资源成本的统一计量口径
- 为恢复评测定义“代码未变”的判定规则
