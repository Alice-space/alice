# ADR 0004: 审核采用版本绑定、固定席位与租约聚合

## 背景

Alice 需要对计划和代码进行多 Agent 审核。若不把审核绑定到明确版本，并等待固定席位收敛，就会出现：

- 旧 verdict 推进新版本
- 首个 reject 或 approve 过早推进状态
- 某个审核 Agent 崩溃导致活锁

## 决策

采用统一审核模型：

- 审核必须绑定 `plan_version` 或 `pr_head_sha`
- 每轮审核先生成 `AuditRequest`
- `AuditRequest` 固定席位、deadline 和 lease
- verdict 收齐、席位过期或 deadline 到达后再聚合
- 前两轮冲突默认按不通过处理
- 第三轮仍冲突则升级人工仲裁

## 备选方案

### 方案 A：任一 reject 立即失败

优点：

- 逻辑简单

缺点：

- 对到达顺序敏感
- 无法表达缺席和迟到 verdict

### 方案 B：审核 Agent 自由协商后给单一结论

优点：

- 对人类界面更简洁

缺点：

- 系统内部不可审计
- 容易隐藏分歧来源

## 影响

- 审核引擎可复用于计划审核与代码审核
- 需要独立的租约检查器
- 需要仲裁对象和人工回流链路

## 后续事项

- 固化 `missing` 席位的聚合语义
- 为仲裁卡片定义固定字段和失效规则
